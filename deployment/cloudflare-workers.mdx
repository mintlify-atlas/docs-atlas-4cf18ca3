## Overview

acmcsuf.com runs on [Cloudflare Workers](https://workers.cloudflare.com/), a serverless execution environment that runs on Cloudflare's global edge network. It is both free and performant for the project's needs.

The site uses the official [`@sveltejs/adapter-cloudflare`](https://github.com/sveltejs/kit/tree/main/packages/adapter-cloudflare) adapter (currently `^7.2.4`) to compile SvelteKit into a Worker-compatible bundle.

## Adapter configuration

In `svelte.config.js`, the Cloudflare adapter is wired in directly:

```javascript
import adapter from '@sveltejs/adapter-cloudflare';

const config = {
  kit: {
    adapter: adapter(),
  },
};
```

This tells SvelteKit to emit a Workers-compatible entry point instead of a Node.js server.

## Build output

Running `vite build` (or `vite build --mode production`) produces the following structure:

```
.svelte-kit/cloudflare/
├── _worker.js          # Worker entry point
├── _app/               # JavaScript/CSS bundles
│   ├── immutable/
│   └── version.json
└── [other static files]
```

`_worker.js` is the file uploaded to Cloudflare Workers. All static assets in the directory are uploaded to Workers Assets and served from Cloudflare's CDN.

<Note>
  A size-limit check enforces that `_worker.js` stays under **50 KB**. This is validated automatically in CI via `npm run test:size`.
</Note>

## `wrangler.toml` configuration

The full configuration file at the project root:

```toml
# For more details on how to configure Wrangler, refer to:
# https://developers.cloudflare.com/workers/wrangler/configuration/

name = "acmcsuf-com"
main = ".svelte-kit/cloudflare/_worker.js"
compatibility_date = "2026-01-04"
compatibility_flags = [ "nodejs_als", "nodejs_compat" ]

[assets]
binding = "ASSETS"
directory = ".svelte-kit/cloudflare"

[observability]
[observability.logs]
enabled = true
head_sampling_rate = 1
invocation_logs = true
persist = true

# ==== Route/Domain Configuration ==================================================================
# https://developers.cloudflare.com/workers/configuration/routing/

# This uses vars defined in .env.production (make sure debug is off!)
[env.production]
routes = [
  { pattern = "acmcsuf.com", custom_domain = true }
]
# https://developers.cloudflare.com/workers/configuration/previews/
# To create a preview deployment, use one of the following:
#   `wrangler versions upload -e [env-name]`
#   `wrangler versions upload -e [env-name] --preview-alias [alias-name]`
# The latter puts the alias name in the URL, useful for tracking an in-progress feature.
# Live preview URLs can be found in the "Deployments" tab of the Worker.
workers_dev = true
preview_urls = true

# More environments can be created here, and they can be deployed by running
# `wrangler deploy -e [env-name]`, and they'll use a `.env.[env-name]` file.
# Example:

# [env.staging]
# route = "staging.acmcsuf.com/*"
# workers_dev = false
```

### Key settings

| Setting | Value | Purpose |
|---|---|---|
| `name` | `acmcsuf-com` | Worker name in the Cloudflare dashboard |
| `main` | `.svelte-kit/cloudflare/_worker.js` | Entry point generated by SvelteKit |
| `compatibility_date` | `2026-01-04` | Cloudflare runtime version pin |
| `compatibility_flags` | `nodejs_als`, `nodejs_compat` | Node.js AsyncLocalStorage + API compatibility |
| `[assets].binding` | `ASSETS` | Binding name used to access static assets in worker code |
| `[assets].directory` | `.svelte-kit/cloudflare` | Path to the compiled static asset output |
| `[observability.logs]` | enabled, persisted | Request logs with 100% head sampling |

## Production vs. preview environments

### Production

The `[env.production]` block in `wrangler.toml` routes traffic for `acmcsuf.com` as a custom domain. Deploying to production runs:

```bash
wrangler deploy -e production
```

### Preview deployments

With `workers_dev = true` and `preview_urls = true`, every version upload creates a versioned preview URL:

```
<VERSION_PREFIX OR ALIAS>-<WORKER_NAME>.<SUBDOMAIN>.workers.dev
```

To upload a preview with a named alias (useful for tracking in-progress features):

```bash
wrangler versions upload -e production --preview-alias feature-name
```

To list all uploaded versions (also visible in the Cloudflare dashboard):

```bash
wrangler versions list -e production
```

Cloudflare supports rolling back to any listed version at any time.

## Adding additional environments

New environments (e.g., staging) can be added by extending `wrangler.toml`:

```toml
[env.staging]
routes = [
  { pattern = "staging.acmcsuf.com", custom_domain = true }
]
workers_dev = true
```

Then create a `.env.staging` file and deploy with:

```bash
vite build --mode staging
wrangler deploy -e staging
```
